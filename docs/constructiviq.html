The Problem We're Solving

The current ciq-client codebase manages permissions through a single 280-line switch statement with hardcoded role arrays. Every permission change requires a code deployment, and there is no concept of per-project scoping, temporary grants, or dependency resolution.

No Admin Dashboard

Permission changes require engineer time and a full deployment cycle. No self-service for project managers.

No Per-Project Scoping

A user's role applies globally. Cannot give someone admin on one project and viewer on another.

No Temporary Grants

Cannot temporarily elevate permissions for a specific task or deadline without code changes.

No Dependency Resolution

Granting "delete" does not auto-grant "read" — leading to broken states where users can delete items they cannot see.

      EUserRoleName[EUserRoleName.gc_project_admin],

    return !roles.some((r) => r === role);
  }

AFTER — Declarative RBAC Engine

// One-line permission check

// => true  (sarah is gc_project_admin)

// => false (excluded via admin override)

// => true  (granted via admin override)

// UI manifest for frontend

// => { permissions: { submittals: { read: true, create: false, ... } } }

ConstructivIQ Domain Model

ConstructivIQ operates on a dual-scope architecture. Permissions are resolved at two distinct levels, each with its own set of roles and resources.

Subscription Level

Organization management scope covering projects, users, settings, companies, and audit logs. Managed by subscription_admin, subscription_editor, and subscription_viewer roles.

Project Level

Construction workflow scope covering submittals, materials, schedules, workflows, and 22 other resources. Scoped per-project with 12 distinct roles.

Scope Hierarchy

Organization (Subscription) +-- Subscription Users & Roles | subscription_admin / subscription_editor / subscription_viewer +-- Companies | Trade partners, architect firms, engineering firms +-- Projects +-- Project Users & Roles | gc_project_admin, gc_foreman, sc_project_admin, architect, owner, ... +-- Submittals --> Workflows --> Approvals +-- Materials --> Date Blocks --> Templates +-- Schedules --> Activities --> Linking +-- Bid Packages / Design Packages +-- Risk Reports / Custom Reports / Saved Views +-- Integrations (ACC, Procore, CMiC)
Key Insight: A single user can hold different roles on different projects. For example, Sarah Chen is gc_project_admin on both project_alpha and project_beta, while Lisa Thompson is architect on project_alpha but gc_sc_architect on project_beta.

Multi-Company Collaboration

Construction projects inherently involve multiple companies working together. The RBAC framework enforces company-scoped access where each organization's users have precisely the permissions their role requires — no more, no less.

General Contractor (GC)

Full project control. Creates and manages submittals, assigns work to SCs, configures workflows, manages all users and integrations.

Subcontractor (SC)

Limited to their scope. Can view and respond to assigned submittals, upload attachments, update material data, but cannot modify project configuration.

Architect

Design review focus. Reviews submittals, approves design workflows, uploads markups. No access to schedules, bid packages, or project configuration.

Owner

Oversight and approval. Read access to key documents, approval authority on submittals and materials, risk report visibility. Cannot create or modify project data.

Admin Overrides — Grants & Exclusions

The RBAC engine supports per-user, per-project overrides that do not require code changes. These are managed via an admin dashboard and stored alongside the user record.

Real ConstructivIQ Scenarios

Exclusion — Restrict Downloads

James Wilson (sc_foreman on project_alpha) has download:materials excluded. He can view materials on screen but cannot export files — required for IP protection on a sensitive project.

Grant — Architect Risk Access

Lisa Thompson (architect on project_alpha) is granted download:risk_reports. Normally architects have no risk report access, but the owner requested Lisa review schedule risk data for a critical submittal.

Exclusion — Restrict User Deletion

Bob (hypothetical team_lead) has delete:project_users excluded. He can manage user assignments but cannot remove anyone from the project — only the GC admin can.

Evaluation Flow — Lisa's Risk Report Access

When the engine evaluates engine.can("lisa", "download", "risk_reports", "project_alpha"):

1. Look up Lisa's role on project_alpha
-->
2. Role = "architect"
-->
3. architect has NO download:risk_reports
-->
4. Check grants: FOUND download:risk_reports
-->
5. ALLOWED

Evaluation Flow — James's Material Download

When the engine evaluates engine.can("james", "download", "materials", "project_alpha"):

1. Look up James's role on project_alpha
-->
2. Role = "sc_foreman"
-->
3. sc_foreman HAS download:materials
-->
4. Check exclusions: FOUND download:materials
-->
5. DENIED
Precedence Rule: Exclusions always override grants. If a permission is both granted and excluded for the same user/project/resource, the exclusion wins. This ensures security constraints are always honored.

The usePermissions Hook

// hooks/usePermissions.ts
import { useContext } from 'react';
import { ProjectContext } from 'contexts/ProjectContext';
import { AuthContext } from 'contexts/AuthContext';

interface PermissionManifest {
  permissions: Record<string, Record<string, boolean>>;
}

export function usePermissions() {
  const { currentUser } = useContext(AuthContext);
  const { currentProject } = useContext(ProjectContext);

  // Manifest is fetched once per project load and cached
  const manifest: PermissionManifest = currentUser.manifests[currentProject.id];

  const can = (resource: string, action: string): boolean => {
    return manifest?.permissions?.[resource]?.[action] ?? false;
  };

  return { manifest, can };
}

// Usage in any component:
const { can } = usePermissions();

if (can('submittals', 'create'))   { /* show create button */ }
if (can('workflows', 'approve'))   { /* show approve action */ }
if (can('schedules', 'manage'))    { /* show schedule admin */ }

API Integration

The backend enforces permissions at the API layer. Since ConstructivIQ uses GraphQL via Hasura with custom resolvers, the RBAC engine integrates at both the middleware and resolver levels.

Express Middleware

// middleware/rbac.ts
import { RBACEngine } from '@constructiviq/rbac';

const engine = new RBACEngine(rbacModel);

export function requirePermission(action: string, resource: string) {
  return async (req: Request, res: Response, next: NextFunction) => {
    const userId = req.user.id;
    const projectId = req.params.projectId;

    const allowed = engine.can(userId, action, resource, projectId);

    if (!allowed) {
      return res.status(403).json({
        error: 'Forbidden',
        required: `${action}:${resource}`,
        project: projectId,
      });
    }

    next();
  };
}

// Route usage:
app.post(
  '/api/projects/:projectId/submittals',
  requirePermission('create', 'submittals'),
  submittalController.create
);

app.delete(
  '/api/projects/:projectId/submittals/:id',
  requirePermission('delete', 'submittals'),
  submittalController.delete
);

GraphQL Resolver Guard (Hasura Action Handler)

// resolvers/submittal.resolver.ts
import { RBACEngine } from '@constructiviq/rbac';

const engine = new RBACEngine(rbacModel);

export const resolvers = {
  Mutation: {
    createSubmittal: async (_, args, context) => {
      // Extract user and project from Hasura session variables
      const userId = context.session['x-hasura-user-id'];
      const projectId = args.project_id;

      // RBAC check
      if (!engine.can(userId, 'create', 'submittals', projectId)) {
        throw new ForbiddenError(
          `User ${userId} cannot create submittals on project ${projectId}`
        );
      }

      // Proceed with creation
      return await SubmittalService.create(args.input, projectId);
    },

    approveWorkflow: async (_, args, context) => {
      const userId = context.session['x-hasura-user-id'];
      const projectId = args.project_id;

      if (!engine.can(userId, 'approve', 'workflows', projectId)) {
        throw new ForbiddenError(
          `User ${userId} cannot approve workflows on project ${projectId}`
        );
      }

      return await WorkflowService.approve(args.workflow_id, userId);
    },
  },

  Query: {
    riskReports: async (_, args, context) => {
      const userId = context.session['x-hasura-user-id'];
      const projectId = args.project_id;

      if (!engine.can(userId, 'read', 'risk_reports', projectId)) {
        throw new ForbiddenError(
          `User ${userId} cannot view risk reports on project ${projectId}`
        );
      }

      return await RiskReportService.list(projectId);
    },
  },
};

Hasura Permission Rules (Declarative)

-- Hasura metadata can reference RBAC via session variables.
-- The manifest is computed server-side and injected into the
-- x-hasura-allowed-actions session variable.

-- Example: submittals select permission
{
  "role": "project_user",
  "permission": {
    "filter": {
      "project_id": { "_eq": "x-hasura-project-id" }
    },
    "columns": ["*"],
    "check": {
      "_exists": {
        "type": "custom_function",
        "name": "rbac_check",
        "arguments": {
          "user_id": "x-hasura-user-id",
          "action": "read",
          "resource": "submittals",
          "project_id": "x-hasura-project-id"
        }
      }
    }
  }
}
Performance Note: The manifest is computed once per session (or per project switch) and cached. Individual engine.can() calls resolve in O(1) from the cached manifest. No database queries are needed at request time.
