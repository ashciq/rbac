Why RBAC? The Problem

Most enterprise applications start with simple permission checks sprinkled throughout the codebase. Over time, this approach creates a tangled web of authorization logic that is expensive to maintain, impossible to audit, and dangerously inconsistent between services.

The Anti-Pattern: Hardcoded Permission Checks

// This is what happens without a proper RBAC framework
function handleRequest(user, action, resource) {
  switch (user.role) {
    case 'admin':
      return true;  // admins can do everything... right?
    case 'manager':
      if (resource === 'reports' && action !== 'delete') return true;
      if (resource === 'orders' && action === 'read') return true;
      if (resource === 'invoices') return action !== 'delete';
      // TODO: what about the new 'analytics' resource?
      return false;
    case 'editor':
      if (resource === 'reports') return action === 'read' || action === 'update';
      // Bug: forgot to add 'orders' access for editors
      return false;
    case 'viewer':
      return action === 'read';
    default:
      return false;  // new roles require code changes + deployment
  }
}

Scattered Logic

Permission checks are duplicated across controllers, services, and middleware. A single policy change means updating dozens of files across multiple repositories.

No Admin Dashboard

Business stakeholders cannot adjust permissions without filing a ticket. Every role change requires a code deployment, creating a bottleneck.

No Audit Trail

When a security incident occurs, there is no way to answer "who had access to what, and when?" Compliance teams cannot produce access reports.

Frontend/Backend Mismatch

The UI shows buttons the user cannot click. The backend rejects actions the UI allowed. Two separate permission systems drift apart over time.

No Temporary Access

There is no mechanism to grant a contractor read access for one week, or to revoke a departing employee's access to sensitive resources without a code change.

Role Explosion

Without inheritance, every unique combination of permissions requires a new role. 10 resources with 6 actions each can produce hundreds of bespoke roles.

The Solution: Centralized, Config-Driven RBAC

// With RBAC framework: one line, any language, any service
const allowed = rbac.can(user, 'approve', 'invoices');

// The JSON config is the single source of truth
// No switch statements. No scattered logic. No drift.
Result: One JSON config drives all permission decisions across every service, every language, and every UI. Changes are reviewed in version control, applied without deployment, and auditable from day one.

JSON Config = Contract

One file defines every resource, action, role, dependency, and user. All SDKs load the same file and produce identical results. Version-controlled, diff-able, reviewable.

Multi-Language SDKs

Reference implementations in Python, TypeScript, and Go. Each SDK implements the same resolution algorithm. Add new languages by following the specification.

Backend Enforcement

Middleware intercepts every API request and calls can(user, action, resource). Unauthorized requests are rejected before reaching business logic.

Frontend Manifest

exportUIManifest(user) produces a flat boolean map. The UI renders show/hide decisions without additional API calls. Same config = same result on both sides.

Core Concepts

11 Standard Actions

Every permission in the system is composed of an action paired with a resource. The framework defines 11 standard actions that cover the full spectrum of enterprise operations.

ActionDescriptionReal-World Example
read View a single resource Open an invoice to view its details
list View a collection of resources Browse the orders table, see all reports
create Create a new resource Submit a new purchase order
update Modify an existing resource Edit a customer profile, change order status
delete Remove a resource Delete a draft report, remove a user account
upload Upload files or attachments Attach a PDF to an invoice, upload a profile photo
download Download files or exports Export a report as CSV, download an attachment
approve Approve or sign off on a resource Approve a purchase order, sign off on an invoice
assign Assign a resource to a user or group Assign a ticket to a team member, delegate a task
archive Archive or soft-delete a resource Archive a completed project, retire an old report
manage Full administrative control Manage user accounts, configure system settings

Resources

Resources are the entities in your system that users interact with. Each resource declares which actions are valid for it. Examples:

orders

read, list, create, update, delete, approve, assign, archive

invoices

read, list, create, update, approve, download, archive

reports

read, list, create, update, delete, download, archive

users

read, list, create, update, delete, assign, manage

settings

read, update, manage

audit_logs

read, list, download

Permission = action:resource

Every permission is expressed as action:resource. This simple, composable format makes permissions readable, searchable, and programmatically verifiable.

// Example permissions
approve:invoices     // Can approve invoices
download:reports     // Can download reports
manage:users         // Full admin control over user accounts
create:orders        // Can create new orders
read:audit_logs      // Can view the audit trail

Permission Dependencies

Some permissions logically require other permissions. The framework resolves these automatically so you never end up with a user who can delete but cannot read, or who can manage but cannot update.

PermissionAutomatically Requires
manage:users read:users + list:users + create:users + update:users
approve:invoices read:invoices + list:invoices
delete:orders read:orders + update:orders
manage:settings read:settings + update:settings
download:reports read:reports + list:reports
archive:orders read:orders + update:orders
Dependency Graph Example: manage:users manage:users | +--- read:users (view individual user profiles) | +--- list:users (browse the user directory) | +--- create:users (add new user accounts) | +--- update:users (modify user profiles and roles) Granting "manage:users" automatically includes all four. You cannot manage what you cannot see or modify. ------------------------------------------------------- Dependency Graph Example: approve:invoices approve:invoices | +--- read:invoices (view invoice details) | +--- list:invoices (browse invoices pending approval) An approver must be able to read what they are approving.

Role Hierarchy

Generic Corporate Role Tree

+-----------+ | ADMIN | +-----+-----+ | +------+-------+ | MANAGER | +--+--------+--+ | | +---------+ +----------+ | | +-----+------+ +---------+---------+ | EDITOR | | USER_MANAGER | +-----+------+ +-------------------+ | +-------+----------+ | CONTRIBUTOR | +-------+----------+ | +-----+------+ | VIEWER | +------------+ Inheritance flows upward: viewer --> contributor --> editor --> manager --> admin user_manager --> manager --> admin Each role inherits ALL permissions from its children.

How Composite Roles Work

A composite role inherits from multiple child roles, combining their permissions. This eliminates role explosion by composing capabilities from building blocks.

Example: manager = editor + user_manager. The manager inherits all content editing permissions from editor AND all user administration permissions from user_manager, without duplicating any permission definitions.

User Management

4-Layer Permission Evaluation

Every permission check passes through four layers in strict priority order. This model is called deny-override: exclusions always win.

1. Exclusion?
DENY
→
2. Grant?
ALLOW
→
3. Role Perms?
ALLOW
→
4. Default
DENY
can(user, action, resource)? | +-- Step 1: Is action:resource in user.exclusions? | YES --> return DENY immediately | NO --> continue | +-- Step 2: Is action:resource in user.grants? | YES --> return ALLOW | NO --> continue | +-- Step 3: For each role in user.roles: | Resolve role + inherited roles --> collect all permissions | Is action:resource in effective permissions? | YES --> return ALLOW | NO --> continue | +-- Step 4: No match found return DENY (default-deny / fail-closed)

Example Users

Normal Role-Based User

Sarah Chen — contributor

Roles: contributor

Grants: none | Exclusions: none

Sarah can read/list dashboards, orders, invoices, and reports. She can create new orders, invoices, and reports. She cannot update, delete, approve, or manage anything. Pure role-based evaluation.

User with Extra Grant

David Park — contributor + grant

Roles: contributor

Grants: download:reports | Exclusions: none

David has all contributor permissions plus the ability to download reports. This was granted by an admin because David serves as an external auditor who needs report exports without having the full editor role.

User with Exclusion

Maria Lopez — manager + exclusion

Roles: manager

Grants: none | Exclusions: delete:users

Maria has full manager permissions, but she is explicitly blocked from deleting user accounts. Even though user_manager (inherited by manager) grants user CRUD, the exclusion overrides the role. This is useful for compliance: some managers should modify users but not remove them.

Multi-Role User

James Wright — contributor + user_manager

Roles: contributor, user_manager

Grants: none | Exclusions: none

James has content creation abilities from contributor AND user administration from user_manager. His effective permissions are the union of both roles. This avoids creating a bespoke "contributor_with_user_admin" role.

Key Principle: Exclusions always override grants and role permissions. This ensures that security constraints are never accidentally bypassed, even when a user is assigned a highly privileged role. When in doubt, the system denies.

Organizational Hierarchy

Every user can optionally have a reports_to field pointing to their manager. This creates an organizational tree that enables hierarchical data access and management chain traversal.

Org Chart with reports_to Chain +------------------+ | Alice Thompson | | CEO / admin | +--------+---------+ | +------------+------------+ | | +--------+--------+ +--------+--------+ | Bob Martinez | | Carol Davis | | VP Ops / mgr | | VP Sales / mgr | +--------+--------+ +--------+--------+ | | +--------+--------+ +--------+--------+ | David Park | | Sarah Chen | | Team Lead | | Sales Lead | +--------+--------+ +--------+--------+ | | +--------+--------+ +--------+--------+ | Eva Garcia | | James Wright | | contributor | | contributor | +-----------------+ +-----------------+

What This Enables

Subordinate Data Access

Bob can view performance data for David, Eva, and anyone else in his reporting chain. He cannot see data for Sarah or James (Carol's chain).

Management Chain Traversal

Given any user, the system can walk up the reports_to chain to find all managers. Eva's chain: Eva → David → Bob → Alice.

Approval Workflows

An expense report submitted by Eva must be approved by her direct manager (David) or anyone above in the chain. The org tree provides the approver list.

Delegation

When Bob is on vacation, Alice can temporarily grant Bob's subordinates to report to Carol. The org tree is config-driven, not hardcoded.

// Org hierarchy queries
engine.getSubordinates("bob_martinez")
// Returns: ["david_park", "eva_garcia"]

engine.getManagementChain("eva_garcia")
// Returns: ["david_park", "bob_martinez", "alice_thompson"]

engine.isManagerOf("bob_martinez", "eva_garcia")
// Returns: true (transitive)

Angular Example

// permission.directive.ts
@Directive({ selector: '[rbacCan]' })
export class PermissionDirective {
  @Input() rbacCan: string;

  constructor(
    private templateRef: TemplateRef<any>,
    private viewContainer: ViewContainerRef,
    private rbac: RbacService
  ) {}

  ngOnInit() {
    if (this.rbac.can(this.rbacCan)) {
      this.viewContainer.createEmbeddedView(this.templateRef);
    } else {
      this.viewContainer.clear();
    }
  }
}

// Usage in template:
// <button *rbacCan="'approve:invoices'">Approve</button>
// <button *rbacCan="'delete:orders'">Delete Order</button>
Important: The UI manifest is a convenience, not a security boundary. The backend must independently enforce permissions on every API request. Never rely solely on frontend permission checks.

Cross-Language SDK Support

The framework provides reference SDK implementations in three languages. All SDKs load the same JSON config and implement the same resolution algorithm, guaranteeing identical behavior regardless of which language your service uses.

Python SDK

Reference Implementation

from rbac import RBACEngine

engine = RBACEngine("rbac_config.json")

# Check permission
if engine.can("sarah_chen", "create", "orders"):
    create_order(data)

# Explain a decision
reason = engine.explain("sarah_chen", "delete", "users")
# "DENY: no role grants delete:users"

# Export UI manifest
manifest = engine.export_ui_manifest("sarah_chen")

# Get subordinates
team = engine.get_subordinates("bob_martinez")

TypeScript SDK

Full-Stack

import { RBACEngine } from '@your-org/rbac';

const engine = new RBACEngine('rbac_config.json');

// Check permission
if (engine.can('sarah_chen', 'create', 'orders')) {
  await createOrder(data);
}

// Explain a decision
const reason = engine.explain('sarah_chen', 'delete', 'users');
// "DENY: no role grants delete:users"

// Export UI manifest
const manifest = engine.exportUIManifest('sarah_chen');

// Middleware integration
app.use(rbacMiddleware(engine));

High-Performance Backend

Unified API Surface

MethodDescriptionReturns
can(user, action, resource) Check if user has permission boolean
explain(user, action, resource) Explain why permission was granted or denied string (human-readable reason)
exportUIManifest(user) Export flat permission map for frontend { [permission]: boolean }

The JSON config is the contract. Every SDK in every language implements the same resolution algorithm against the same file. Switching languages or adding a new microservice requires zero permission logic changes — just load the config.

Database Support

For organizations that need to persist RBAC configuration in a database rather than (or in addition to) a JSON file, the framework provides schema migrations and adapters for three major databases.

PostgreSQL

Recommended

Full relational schema with foreign key constraints, row-level security policies, and optimized indexes for permission lookups. Supports JSONB columns for flexible metadata storage.

-- Initialize schema
psql -f rbac_schema.sql

-- Query effective permissions
SELECT * FROM effective_permissions
WHERE user_id = 'sarah_chen';

-- Audit log
SELECT * FROM permission_audit_log
ORDER BY timestamp DESC LIMIT 50;

MySQL

Widely Supported

Compatible schema with InnoDB engine for transactional integrity. Includes stored procedures for common permission queries and optimized compound indexes.

-- Initialize schema
mysql -u root < rbac_schema.sql

-- Check permission via stored proc
CALL check_permission('sarah_chen',
  'create', 'orders');

-- List all roles for a user
SELECT r.name FROM roles r
JOIN user_roles ur ON r.id = ur.role_id
WHERE ur.user_id = 'sarah_chen';

MongoDB

Document Store

Document-based schema that maps naturally to the JSON config format. Embedded subdocuments for roles and permissions. Aggregation pipelines for complex permission queries.

// Initialize collections
mongosh < rbac_init.js

// Query user permissions
db.users.findOne(
  { _id: "sarah_chen" },
  { permissions: 1, roles: 1 }
);

// Aggregate effective permissions
db.users.aggregate([
  { $match: { _id: "sarah_chen" } },
  { $lookup: { from: "roles", ... } }
]);

Industry Use Cases

The RBAC framework is industry-agnostic by design. Below are examples of how it applies across different verticals, each with their own resources, compliance requirements, and organizational structures.

Healthcare

HIPAA

Resources: patient_records, prescriptions, lab_results, appointments, billing

Key Roles: physician, nurse, admin_staff, billing_clerk, pharmacist

Compliance: HIPAA requires strict audit trails and minimum-necessary access. Exclusions ensure a billing clerk cannot read clinical notes. The explain() method provides the audit trail regulators require.

Finance

SOX

Resources: transactions, accounts, reports, audit_trails, compliance_docs

Key Roles: analyst, trader, risk_manager, compliance_officer, auditor

Compliance: SOX mandates separation of duties. A trader who can execute transactions must not approve their own trades. Exclusions enforce this segregation. The permission matrix is version-controlled for audit evidence.

Construction

Multi-Company

Resources: submittals, rfis, materials, daily_logs, contracts, drawings

Key Roles: project_manager, superintendent, subcontractor, inspector, owner

Compliance: Multi-company workflows where subcontractors see only their scope. Org hierarchy maps general contractors to subs. Approval chains for submittals follow the reports_to tree.

SaaS / Multi-Tenant

Scalable

Resources: workspaces, projects, integrations, billing, api_keys, features

Key Roles: owner, admin, member, guest, billing_admin

Compliance: Subscription tiers map to role templates. Free-tier users get viewer roles; enterprise users get full admin. Feature flags are just permissions: read:advanced_analytics is gated by plan.

E-Commerce

PCI-DSS

Resources: orders, products, customers, inventory, promotions, payments

Key Roles: store_manager, merchandiser, support_agent, warehouse_staff, finance

Compliance: PCI-DSS requires that access to payment data is strictly limited. Only the finance role can read:payments. Support agents can view orders but not payment details.

Education

FERPA

Resources: courses, grades, student_records, assignments, transcripts, schedules

Key Roles: instructor, teaching_assistant, student, registrar, department_head

Compliance: FERPA protects student records. Only the registrar and assigned instructors can access grades. Teaching assistants can grade assignments but not view full student records. Org hierarchy maps department heads to instructors.

Security Considerations

Server-Side Enforcement

The UI manifest is a convenience layer for rendering. Every API endpoint must independently call can(user, action, resource). The frontend should never be the sole enforcer of permissions. If someone bypasses the UI, the backend still blocks unauthorized actions.

Default Deny (Fail-Closed)

If no rule explicitly grants a permission, the answer is DENY. There is no "allow by default" mode. This means a misconfigured role or missing permission entry results in blocked access, not unauthorized access. Fail-closed is always safer than fail-open.

Cycle Detection at Load Time

When the engine loads a config, it validates the role inheritance graph for cycles before any permission checks occur. A cycle (e.g., role A inherits from role B, which inherits from role A) causes a CyclicDependencyError at startup, not at runtime.

Config-as-Code

The JSON config lives in version control alongside the application code. Every change is reviewed in a pull request, approved by security stakeholders, and traceable via git history. No shadow permissions. No undocumented overrides.

Audit Trail via explain()

Every permission decision can be explained: "ALLOW: role manager grants approve:invoices" or "DENY: user exclusion blocks delete:users". This powers compliance reports, incident investigations, and access reviews.

Exclusions Override Everything

The deny-override model ensures that if an exclusion exists, no grant or role can circumvent it. This is critical for separation of duties, compliance holds, and emergency access revocation.

Security Validation Pipeline (at config load time) Load JSON Config | +-- Validate against JSON Schema | FAIL --> reject config, keep previous | +-- Check for unknown roles in user assignments | FAIL --> reject config | +-- Check for unknown resources in permissions | FAIL --> reject config | +-- Detect cycles in role inheritance graph (DFS) | FAIL --> CyclicDependencyError | +-- Detect cycles in permission dependency graph | FAIL --> CyclicDependencyError | +-- Resolve all role inheritance chains | +-- Resolve all permission dependencies | +-- Engine ready: all checks pass in O(1) via lookup
Never trust the client. The UI manifest tells the frontend what to show. The backend middleware tells the server what to allow. Both read the same config, but only the backend is the security boundary.
