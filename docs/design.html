RBAC Permission Framework

A cross-language, hierarchical Role-Based Access Control system with permission dependencies, composite roles, admin overrides, and UI-exportable manifests.

Overview

This framework provides a centralized, language-agnostic permission management system that can be shared across backend services, frontend applications, and mobile clients. A single JSON configuration file defines all resources, actions, roles, permission dependencies, and users — any SDK implementation loads and resolves the same config identically.

Resource-Action Matrix

Every permission is action:resource. Resources define which actions are valid (read, create, update, delete, upload, manage). Roles grant specific actions on specific resources.

Permission Dependencies

Permissions can require other permissions. manage:users auto-requires read:users, create:users, update:users. Dependencies are resolved transitively.

Composite Roles

Roles inherit from multiple parent roles. team_lead = editor + user_manager. Inheritance forms a DAG with cycle detection.

Admin Overrides

Per-user exclusions strip permissions even if roles grant them. Per-user grants add permissions beyond roles. Exclusions always win (deny-override model).

Org Hierarchy

Users have a reports_to chain. Enables organizational tree traversal, subordinate lookups, and hierarchical permission scoping.

UI-Exportable Manifest

Export a flat JSON permission map per user. Frontend renders show/hide decisions without hitting the backend. Same config, same result, both sides.

Priority Source Result Description
1 (Highest) User Exclusion DENY Admin removes permission for specific user. Overrides everything.
2 User Grant ALLOW Admin adds permission for specific user, beyond their roles.
3 Role Permission ALLOW User's assigned role(s) include this permission (direct or inherited).
4 (Lowest) Default DENY No source grants the permission.

Resolution Flowchart

can(user, action, resource)? | +-- Is action:resource in user.exclusions? | YES -> return DENY | +-- Is action:resource in user.grants? | YES -> return ALLOW | +-- For each role in user.roles: | Is action:resource in role.effective_permissions? | YES -> return ALLOW | +-- return DENY (default)

Permission Dependencies

When a permission is granted, its dependencies are automatically included. This prevents inconsistent states (e.g., a user who can delete but can't read).

PermissionAutomatically Requires
manage:users read:users + create:users + update:users

manage:settings read:settings + update:settings
manage:dashboard read:dashboard
upload:files read:files + create:files
delete:audit_logs read:audit_logs
Dependency Graph Example: manage:users manage:users | +--- read:users (view user list) | +--- create:users (add new users) | +--- update:users (modify user profiles) Granting "manage:users" automatically includes all three. You cannot manage what you cannot see or modify.

User Management

User Configuration

Each user is defined with roles, per-user grants, per-user exclusions, and org hierarchy position.

{
  "bob": {
    "display_name": "Bob Smith",
    "roles": ["team_lead"],              // Assigned roles
    "grants": {},                         // Extra permissions beyond roles
    "exclusions": {
      "users": ["delete"]                // Admin strips this even though team_lead has it
    },
    "reports_to": "alice"                // Org hierarchy
  }
}

Resolution Engine

How Permissions Are Resolved

All resolution happens eagerly at load time. Runtime checks are O(1) set lookups.

LOAD TIME (once): ================== Step 1: Build permission dependency graph manage:users -> {read:users, create:users, update:users} Step 2: For each role, resolve inheritance (DFS with cycle detection) team_lead.inherits = [editor, user_manager] -> editor.inherits = [contributor] -> contributor.inherits = [viewer] -> viewer = {read:dashboard, read:reports, ...} Step 3: Merge all inherited + direct permissions Step 4: Expand permission dependencies for every permission team_lead has manage:dashboard -> auto-add read:dashboard Step 5: Store as effective_permissions (frozen set per role) RUNTIME (per check): ==================== can(user, action, resource): exclusions[resource].has(action)? -> DENY grants[resource].has(action)? -> ALLOW any role.effective[resource].has(action)? -> ALLOW default -> DENY

Cycle Detection

Both permission dependencies and role inheritance use DFS with ancestor tracking. If a node is encountered that's already in the current DFS path, a CyclicDependencyError is raised at load time, preventing invalid configs.

Diamond Dependencies Are OK: If role A inherits B and C, and both B and C inherit D, D's permissions appear once in the result (set union). The cycle detector correctly distinguishes diamond shapes (valid) from true cycles (invalid).

UI Integration

Frontend Permission Manifest

The engine exports a flat JSON manifest per user. Frontend apps use this to conditionally render UI elements without calling the backend for each check.

// Exported by: engine.export_ui_manifest("bob")
{
  "user": "bob",
  "display_name": "Bob Smith",
  "roles": ["team_lead"],
  "permissions": {
    "dashboard":  { "read": true,  "manage": true },
    "reports":    { "read": true,  "create": true,  "update": true,  "delete": true,  "upload": true },
    "projects":   { "read": true,  "create": true,  "update": true,  "delete": true },
    "users":      { "read": true,  "create": true,  "update": true,  "delete": false, "manage": false },
    "settings":   { "read": false, "update": false, "manage": false },
    "files":      { "read": true,  "create": true,  "delete": true,  "upload": true },
    "audit_logs": { "read": true,  "delete": false }
  }
}

Frontend Usage (Any Framework)

// React example
function ReportActions({ manifest }) {

}

Use Cases

1. New Employee Onboarding

Assign viewer role. They get read access to dashboard, reports, projects, files. No config changes needed — just set role.

2. Promote to Contributor

Change role to contributor. Auto-inherits all viewer permissions plus create/update on reports, projects, and file upload. Dependencies auto-resolved.

3. Restrict Specific User

Bob is team_lead but shouldn't delete users. Add exclusions: {"users": ["delete"]}. Exclusion overrides role permission. No role change needed.

4. Grant One-Off Permission

Carol (editor) needs to see audit logs for an investigation. Add grants: {"audit_logs": ["read"]}. No role change, scoped to this user.

5. Multi-Role User

Grace needs both content creation and user management. Assign both contributor and user_manager roles. Permissions merge via set union.

6. Auditor / Compliance

Frank needs read-only access everywhere including audit logs and settings. The auditor role inherits viewer + adds read on users, settings, audit_logs. Cannot modify anything.

7. Manager Sees Team Data

Bob can call get_subordinates("bob") to find Carol, Dave, Grace, Eve. Backend filters data to show records belonging to these users.

8. UI Rendering

On login, backend calls export_ui_manifest(user) and sends the flat permission map. Frontend shows/hides buttons, menus, and pages based on true/false values. Zero additional API calls.

Cross-Language Support

The Contract

Every SDK must implement these guarantees when loading rbac_model.json:

  1. Validate the config against rbac_schema.json
  2. Resolve permission dependencies transitively (DAG, detect cycles)
  3. Resolve role inheritance transitively (DAG, detect cycles)
  4. Compute effective permissions per role (inheritance + dependencies merged)
  5. Evaluate can(user, action, resource) with the precedence: exclusion > grant > role > deny
  6. Export the same UI manifest format for frontend consumption

Security Considerations

Server-Side Enforcement

The UI manifest is for rendering convenience only. Always enforce permissions server-side. The frontend can be bypassed; the backend cannot.

Config Integrity

The rbac_model.json file should be version-controlled, code-reviewed, and deployed through CI/CD. Changes to permissions are code changes.

Fail-Closed (Default Deny)

If a permission is not explicitly granted through any mechanism, the default is DENY. Unknown actions, unknown resources, unknown roles all result in denial.

Cycle Protection

Cyclic dependencies in roles or permissions are detected at load time and raise CyclicDependencyError. The system cannot start with an invalid config.

Audit Trail

The explain() method provides human-readable audit of every permission decision, traceable to the exact role, grant, or exclusion that caused it.

Least Privilege

The role hierarchy is designed so each level adds only what's needed. Exclusions provide a safety valve to remove even inherited permissions for specific users.

SDK Implementations

Every SDK loads the same rbac_model.json and implements identical resolution logic. The output of can() and exportUIManifest() is guaranteed to match across languages.

Python SDK Reference

Full implementation with RBACEngine, dependency resolution, cycle detection, UI export. Zero dependencies beyond stdlib.

from python.engine import RBACEngine

engine = RBACEngine("schema/rbac_model.json")
engine.can("bob", "delete", "users")  # False
engine.export_ui_manifest("bob")      # JSON dict

TypeScript SDK Browser + Node

Full TypeScript implementation with type safety. Works in browser (load config via fetch) and Node.js (load from filesystem).

import { RBACEngine } from './rbac-engine';

// Node.js
const config = JSON.parse(fs.readFileSync('rbac_model.json'));
const engine = new RBACEngine(config);

// Browser
const resp = await fetch('/api/rbac-config');
const engine = RBACEngine.fromJSON(await resp.text());

engine.can('bob', 'delete', 'users');  // false
engine.exportUIManifest('bob');         // UIManifest

Go SDK Backend

Idiomatic Go implementation with exported types and error returns. Suitable for middleware integration in HTTP servers.

Database Integration

Production deployments store RBAC data in a database instead of JSON files. Scripts are provided for PostgreSQL, MySQL, and MongoDB — each with schema, seed data, and helper functions that mirror the SDK API.

Database Schema (All Databases)

# Usage
SELECT can_user('bob', 'delete', 'users');          -- FALSE (excluded)
SELECT can_user('alice', 'manage', 'users');         -- TRUE
SELECT * FROM get_effective_permissions('bob');       -- all perms with source
SELECT export_ui_manifest('bob');                     -- JSON for frontend

PostgreSQL Features: Stored functions (can_user(), export_ui_manifest()), recursive CTEs for role inheritance and org hierarchy, JSONB export for UI manifests, and a materialized audit view.
